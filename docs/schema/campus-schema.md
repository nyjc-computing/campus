# README


| SN | Rule |
| --- | --- |
| 1.0 | Endpoints are hierarchically structured: resource, subresource, ..., verb |
| 2 | Order of endpoint segments should be consistent across multiple interfaces |
| 3 | Object ids, where relevant, are specified after the label |
| 4 | Standard verbs map to HTTP methods: new (POST), get (GET), update (PATCH), delete (DELETE) |
| 5 | Standard extensions to the above verbs:
- add: iimplies a resource already exists and is being added elsewhere
- remove: implies the resource is removed from the current location but still exists (elsewhere) |
| 6 | - object names and custom verbs should not reuse or redefine these standard verbs |
| 7 | - path segments in plural imply resources (usually mapping to table rows or collection documents, e.g. users instead of user) |
| 8 | - path segments in singular imply a service (e.g. emailotp) |
| 9 | - avoid ambiguity as far as possible (e.g. data as a name isn't clear) |
| 10 | - end path segments (segment at the end of the path) that are nouns imply resources, possibly aliases (e.g. members, or profile). The GET verb may be omitted if a single row or document is implied; GET must be present in the path if this is not the case (e.g. the end segment is a plural resource, such as users) |
| 11 | - end path segments (segment at the end of the path) that are verbs (other than get) imply mutation |
| 12 | Events are only emitted on data changes (any verb other than GET), to reduce event volume. GET requests will be logged separately. |
|  | Access |
| 14 | Each access type has a power-of-2 value associated with it |
| 15 | Each circle defines the access permissions of its subcircles. |
| 16 | The access permission of a subcircle is the sum of its target access type values. |
| 17 | All subcircle members share the same access permissions. |
| 18 | If a user is a member of more than one subcircle, their access permission is the highest of the access permissions of their subcircles. |
